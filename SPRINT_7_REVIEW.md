## Verdict: NOT APPROVED — 4 Critical Gaps Must Be Resolved

The Sprint 7 Memory System has a solid domain model and correct architectural foundations. However, four critical gaps mean the system as implemented **does not actually personalize the agent**: memories are loaded but never read by downstream nodes, embeddings are never generated for stored memories, procedural memories have no creation path, and the consolidation pipeline has no JSON output guarantee. These are not design flaws — they are missing wires that prevent the system from functioning.

## Eight Evaluation Questions — Answered First

### Q1: Does memory retrieval actually improve agent decisions?

**Answer: No. The memory is loaded but never consumed.**

The `context_builder_node` (updated in Day 7) loads recent episodes, semantic memories, and procedural patterns, then places them in `memory_context` (a string field in SupervisorState). However, **none of the downstream LLM nodes read `memory_context`**:

- `intent_router_node`: Only reads `user_message`. Does not inject memory into the LLM prompt.
- `task_planner_node`: Not shown updated. The TaskPlanner from Sprint 6 does not accept memory context.
- `result_synthesizer_node`: Reads `user_profile` for "Hi {name}" but does not read `memory_context`.

The memory retrieval infrastructure is complete, but the consumption side is entirely missing. The memory is loaded, stored in state, and silently ignored.

**Fix:** Every LLM-calling node must prepend `memory_context` to its system or user prompt. Example for intent_router:
```python
memory_context = state.get("memory_context", "")
if memory_context:
    user_prompt = f"Context about this user:\n{memory_context}\n\nUser message: {user_message}"
```

### Q2: Is memory context too large?

**Answer: There's a risk but it's manageable with token counting.**

The retrieval loads up to 20 episodes + 10 semantic + 3 procedural. Each episode summary is ~150 chars. Each semantic content is ~200 chars. Rough estimate: 20×150 + 10×200 + formatting = ~5,000 characters ≈ 1,200 tokens. This is well within the 8K budget. However:

- There's no actual token counting (tiktoken or equivalent). The `token_budget` parameter is passed but the trimming logic is primitive (`[:150]` string slicing).
- Episodic summaries can be up to 500 chars (the `final_response[:500]` in the entity factory), blowing the estimate to 20×500 = 10,000 chars ≈ 2,500 tokens for episodes alone.

**Recommendation:** Use `tiktoken` for accurate token counting. Trim memory context to fit within a hard budget of 2,000 tokens for memories (leaving 6,000 for the rest of the prompt).

### Q3: Is the consolidation pipeline reliable?

**Answer: No. Three failure modes.**

1. **No JSON output guarantee.** The consolidation LLM call does not use `response_format={"type": "json_object"}` or structured output mode. The LLM can return text with JSON embedded, or malformed JSON, or an empty response. `json.loads()` on any of these raises and all insights are dropped.

2. **No idempotency.** If the consolidation task runs twice for the same user (Celery retry, duplicate beat firing), the same episodes are processed twice and duplicate semantic memories are created (the UPSERT-by-subject check catches exact matches, but LLM-generated subjects vary slightly between runs).

3. **No error recovery for partial failures.** If 5 of 10 insights succeed and the 6th raises, the transaction commits the first 5 and silently drops the rest. There's no rollback to a safe state.

### Q4: Will semantic memories drift over time?

**Answer: Yes, moderately. Two drift vectors exist.**

1. **Subject variation drift.** The UPSERT key is `subject` (a free-text string generated by the LLM). "User prefers remote roles" and "User likes working remotely" produce different subjects and two separate memories. Over months, a user will accumulate 5-10 near-duplicate memories about the same preference.

2. **Confidence inflation.** Each consolidation run adds evidence episodes even if the LLM is just restating the same fact. `update_evidence` adds +0.03 confidence per episode. After 30 consolidation runs restating the same fact, confidence reaches 1.0 regardless of factual accuracy.

**Fix:** Use embedding similarity on subjects for UPSERT matching. Cap confidence at 0.95 (absolute certainty is impossible from behavioral signals). Require at least one explicit user confirmation to exceed 0.85.

### Q5: Are procedural memories useful?

**Answer: No. They are defined but orphaned.**

The domain model for `ProceduralMemory` is well-designed. The `record_execution()` method correctly implements a running success rate. But **zero code paths create or update procedural memories**:

- The consolidation pipeline only creates SemanticMemory, never ProceduralMemory
- No tool or agent node calls `procedural_repo.save()` or `.record_execution()`
- The `find_by_context()` query always returns an empty list

ProceduralMemory is a V1 feature that was included in Sprint 7's schema but never wired.

**Recommendation:** Accept this as deferred scope. Add a comment in the consolidation pipeline: `# TODO(V1): Extract ProceduralMemory patterns from repeated successful workflows`. Remove the unused retrieval from the context builder to avoid confusing developers.

### Q6: Are embeddings generated correctly?

**Answer: No. Embeddings are never generated for stored memories.**

The `MemoryRetrievalService` generates a query embedding (`self._embedder.generate_embedding(query_text)`) to search against stored memories. But the stored memories themselves have `embedding=None`:

- `EpisodicMemory.record_agent_execution()` does not generate an embedding for the episode
- `SemanticMemory.create_fact()` does not generate an embedding
- The consolidation pipeline does not generate embeddings after creating semantic memories
- There is no Celery task to backfill embeddings

The HNSW indexes from migration 007 are created on empty columns. The cosine similarity search on `embedding` with `is_not(None)` filter returns zero results because all embeddings are NULL.

**Fix:** Add embedding generation as the final step of the consolidation pipeline. After creating semantic memories, generate their embeddings via the DeepSeek embedding API and store them. For episodic memories, generate embeddings at creation time (or batch-generate in the cleanup job).

### Q7: Is memory growth controlled?

**Answer: Partially. Episodic growth is controlled. Semantic and procedural are unbounded.**

- **Episodic:** ✅ Cleanup job deletes rows where `expires_at < NOW()`. Normal episodes get 90-day TTL. But `expires_at` is set to `None` for all episodes — the `record_agent_execution()` factory doesn't set it. The cleanup job will delete nothing.

- **Semantic:** ❌ No expiry, no archival, no pruning. Grows indefinitely. At 10 insights/user/day for 100K users, that's 1M rows/day = 365M rows/year.

- **Procedural:** ❌ Same — unbounded growth.

**Fix:** Set `expires_at = created_at + timedelta(days=90)` in the episodic factory. Add a `deleted_at` soft-delete to SemanticMemory and a Celery task to deactivate memories with `access_count=0` for >180 days.

### Q8: Will this scale to 100k users?

**Answer: Consolidation won't. Retrieval will.**

- **Retrieval:** ✅ HNSW index on 3072d vectors. Three indexed queries per agent invocation. Scales to 100K concurrent reads.

- **Consolidation:** ❌ Sequential per-user processing. For 10K daily active users: 10,000 users × 2 seconds (LLM + DB) = 20,000 seconds = 5.5 hours. This exceeds the 24-hour window and blocks the next day's run. Need batch LLM processing (100 users per call) or parallel Celery workers.

## Detailed Audit Findings

### 1. Episodic Memory Design — B

**What's right:** Immutable append-only records. Type-safe episode categories. Factory methods for common events. Importance scoring at creation time.

**CRIT-1: Episodic embedding never generated (see Q6).** The `embedding` field is always `None`. Vector search over episodes returns nothing.

**MAJ-1: `expires_at` never set.** The entity factory doesn't compute `expires_at`. The cleanup job will find zero expired records.

### 2. Semantic Memory Design — B+

**What's right:** Evidence tracking with versioning. Confidence scoring. Active/inactive soft-delete. UPSERT logic for deduplication.

**CRIT-2: Semantic embedding never generated (see Q6).** Same issue — `embedding=None` for all semantic memories.

**MAJ-2: Subject-based dedup is fragile (see Q4).** String equality on LLM-generated subjects. Two phrasings of the same fact → duplicates.

### 3. Procedural Memory Design — C

**What's right:** Well-designed entity. Running success rate calculation. Active/inactive toggle.

**CRIT-3: Orphaned — zero creation paths (see Q5).** The entire ProceduralMemory subsystem has no code that writes to it. It's a schema without a feature.

### 4. Consolidation Pipeline — C+

**What's right:** Clean 3-step pipeline: fetch → LLM extract → UPSERT. Batch processing of episodes.

**CRIT-4: No JSON output guarantee (see Q3).** `json.loads()` on unvalidated LLM output.

**MAJ-3: Single-user focus ignores batch efficiency.** Each user gets one LLM call. For 100 users all with similar profiles, that's 100 separate LLM calls with nearly identical system prompts. Batch consolidation (10 users per LLM call) would be 10× cheaper.

**MIN-1: `SemanticMemoryType` in the LLM prompt lists types that don't match the enum values.** The prompt says `"type": "profile_fact|skill_knowledge|learned_insight|preference_fact"` but the enum also has `career_narrative` and `general_knowledge`. Inconsistency between prompt and code.

### 5. Memory Retrieval Quality — D

**CRIT-5: Retrieved memories never consumed by agent (see Q1).** The `memory_context` string is populated in state but no LLM node reads it. The retrieval infrastructure is complete but disconnected from the agent's reasoning.

### 6. Memory Ranking Strategy — C+

**What's right:** Importance-based filtering (`min_importance=0.2`). Recency ordering for episodes. Success rate ordering for procedural.

**MAJ-4: No composite ranking score.** Semantic retrieval uses only cosine similarity. It should combine similarity × importance × recency for a composite ranking, similar to the MEMORY.md design: `score = 0.4×similarity + 0.3×importance + 0.2×recency + 0.1×access_freq`.

### 7. Importance Scoring — B

**What's right:** Type-based defaults. Payload-based boosts. Decay function for non-critical memories.

**MIN-2: `ImportanceCalculator` is defined but never called.** The `record_agent_execution()` factory hardcodes `importance=0.5` instead of calling `ImportanceCalculator.calculate(EpisodeType.AGENT_INVOCATION, payload=...)`.

### 8. Memory Decay Strategy — B-

**What's right:** `ImportanceScore.decay()` method with exponential decay. Critical memories (0.9+) exempt.

**MIN-3: Decay function is defined but never applied.** No code iterates over episodic memories and calls `.decay()`. The importance score in the database stays at its initial value forever.

### 9. Memory Versioning — B

**What's right:** `SemanticMemory.version` incremented on content update. Evidence tracking via episode list.

**MIN-4: No version history table.** The current version overwrites the previous content. If consolidation produces a wrong fact that overwrites a correct one, there's no rollback. The `semantic_memory_versions` table from DATABASE.md §3 was not implemented.

### 10. Memory Conflict Resolution — C

**What's right:** UPSERT by subject avoids exact duplicates.

**What's missing:** No detection of contradictory facts. If consolidation run #5 says "User prefers onsite" and consolidation run #10 says "User prefers remote", both coexist with no conflict flag. A downstream agent reading both gets contradictory signals.

### 11. Agent Integration — D

**CRIT-5 (duplicate): Memories loaded but never consumed.**
**MAJ-5: `memory_context` field is a raw string with no structure.** Downstream LLM nodes need to parse an unstructured text block to find relevant information. A structured format (JSON with typed memories) would be more useful for prompt engineering.

### 12. pgvector Design — C+

**What's right:** Separate dimension sizes for episodic (1536d) and semantic (3072d). HNSW indexes with appropriate m and ef_construction parameters. Cosine distance operator.

**CRIT-1/CRIT-2 (duplicate): Embeddings never generated.**
**MIN-5: No `ef_search` configuration at query time.** HNSW `ef_search` controls search recall. Default is 40. For semantic search where recall matters, this should be set to 100+ at query time: `SET LOCAL hnsw.ef_search = 100;`

### 13. HNSW Index Design — B

**What's right:** Correct m=16/ef=200 for 3072d semantic. Correct m=12/ef=150 for 1536d episodic. Both are standard, well-tested configurations.

**MIN-6: Index creation uses `IF NOT EXISTS` but was already created in migration 001.** Migration 001 (`001_initial_schema.py`) already creates `idx_episodic_embedding` and `idx_semantic_embedding` as HNSW indexes. Migration 007 attempts to recreate them. This will fail with "relation already exists" unless `CONCURRENTLY` is used.

### 14. Celery Consolidation Jobs — C+

**What's right:** Separate tasks for consolidation and cleanup. Beat schedule at off-peak hours.

**MAJ-6: Sequential user processing won't scale (see Q8).** 10K users × 2s = 5.5 hours.

**MAJ-7: No task timeout.** If one user's LLM call hangs, the entire consolidation task blocks. No `time_limit` or `soft_time_limit` on the Celery task.

### 15. Cleanup Jobs — C

**MAJ-1 (duplicate): Nothing to clean up because `expires_at` is always NULL.**
**MIN-7: No semantic memory pruning.** Inactive semantic memories accumulate forever.

### 16. Security — B

**What's right:** Memory queries always scoped to `user_id`. No cross-user memory access. Memory context endpoint requires authentication.

**MIN-8: `GET /v1/agent/memory/context` returns all facts without pagination.** For a long-tenured user with 500+ semantic memories, this could return a large payload. Add cursor pagination.

### 17. Scalability — C

See Q8.

### 18. Cost Efficiency — C+

**What's right:** Daily consolidation (not 6-hourly). LLM calls only for users with new episodes.

**MAJ-3 (duplicate): No batch consolidation.**

**MIN-9: Episodic embedding generation (when fixed) will be expensive.** 3072d embeddings for every episodic memory × 500K episodes/day = significant API costs. Consider generating embeddings only for high-importance episodes (≥0.5).

### 19. Testing Coverage — C+

**What's right:** Entity tests cover core behaviors. Importance calculator tests are present. Memory API integration test exists.

**What's missing:**
- No consolidation pipeline integration test
- No test verifying memory context is consumed by agent nodes
- No test verifying embeddings are generated and searchable
- No test for cleanup job
- No test for procedural memory (since it's never created)

## Memory System Best Practice Recommendations

1. **Generate embeddings at write time, not read time.** Add embedding generation to the consolidation pipeline's final step. Use the DeepSeek embedding API (already available from Sprint 3's `DeepSeekClient.generate_embedding()`).

2. **Use structured JSON output mode for consolidation.** DeepSeek supports `response_format={"type": "json_object"}`. This guarantees valid JSON. Always use it for structured extraction.

3. **Implement composite retrieval ranking.** Combine vector similarity (0.4) + importance (0.3) + recency (0.2) + access frequency (0.1). This matches the MEMORY.md design and produces better results than pure cosine similarity.

4. **Batch consolidate users.** Group 20 users with unconsolidated episodes into a single LLM call. Prompts for each user separated by clear markers. 10× cost reduction.

5. **Set Celery task timeouts.** `time_limit=3600` (1 hour) and `soft_time_limit=3300` (55 min) on the consolidation task. This prevents a hung LLM call from blocking the worker indefinitely.

6. **Wire memory context into every LLM node.** The intent router, task planner, and result synthesizer should all receive `memory_context` in their prompts. This is the entire point of the memory system.

7. **Defer procedural memory to V1.** Remove the unused retrieval from the context builder. Keep the schema for forward compatibility. Focus Sprint 7 on making episodic + semantic work end-to-end.

8. **Cap semantic memory confidence.** Add `confidence = min(0.95, confidence)` in `update_evidence()`. Absolute certainty is a smell.

## SPRINT 7 — NOT APPROVED

**Rationale:** The memory system is architecturally correct but functionally inert. It captures memories, stores them, and retrieves them — but the agent never reads them, the vector search returns nothing because embeddings are never generated, procedural memory has no creation paths, and the consolidation pipeline can silently fail on malformed LLM output. These are wiring issues, not design issues. The architecture is sound; the implementation needs 8 hours of targeted fixes to bridge the gaps.

**Required before resubmission:** Fix CRIT-1 through CRIT-5. After these 5 fixes, the system will: generate embeddings → return vector search results → load memories into agent context → agent consumes memories in prompts → consolidation reliably produces structured output. That's the minimum viable memory system.
